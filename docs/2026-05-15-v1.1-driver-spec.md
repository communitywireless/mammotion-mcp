## The bug fixed

`mow_area` v1.0 returns "success" after the HA service-call sequence (cancel_job + start_stop_blades + start_mow) completes 200 OK + state briefly transitions to "mowing" within 60s. Empirical failure 2026-05-15 09:25 HST: state held "mowing" for exactly 24 seconds → paused. work_area never reached the target area hash. blade_used_time unchanged (166.49 hr). Mower never physically mowed. Tool returned success — W-003 "test the faucet, not the plumbing" violation.

## The fix — Investigator-spec'd 3-phase verification

Per the Investigator report §5, add a `_verify_mowing(target_area_hash, target_area_name)` helper that runs AFTER step 3 (start_mow) and BEFORE returning success. Three phases:

| Phase | Window | Success criterion | Failure interpretation |
|---|---|---|---|
| 1 | 0–90s | `state=mowing` + `activity=MODE_WORKING` SUSTAINED ≥30s | Mower aborted before sustained mowing (preflight failure / dock return) |
| 2 | 90–600s | `sensor.luba2_awd_1_work_area` contains `"area <target_hash>"` | Mower never reached the target area within 10 min |
| 3 | 600–1800s | `sensor.luba2_awd_1_blade_used_time` delta ≥ 0.001 hr (~3.6 sec of blade time) | Blades never physically engaged (SysReport lag: 5–17 min typical, so 20-min window) |

Phase 3 is THE proof — blade_used_time is the only signal that conclusively proves blades physically spun. `current_cutter_rpm` does NOT exist in HA's surface (Investigator §1).

## protocol_version bump

Increment `protocol_version` from 1 → 2 in v1.1 mow_area responses. Per hyperscale-ready-disciplines.md #3, callers can detect verification semantics. Other tools (status, list_areas, etc.) keep `protocol_version: 1` until they get a v1.1-equivalent change.

## Edge cases the Driver MUST handle

1. **Stale-SysReport rollback** (Investigator §2, signal 2): mid-Phase-3, current_value < baseline. Per pseudocode: `if blade_delta < 0: continue` (do not update baseline, keep polling). Test this case explicitly.

2. **mow_duration_sec interaction**: If caller sets `mow_duration_sec=300`, the existing v1.0 behavior was: sleep 300s, then dock. v1.1 verification runs IN PARALLEL with mow_duration_sec sleep. Concretely: if verify=True AND mow_duration_sec set, verification runs first (3-phase, up to 30 min); if it succeeds Phase 3 BEFORE mow_duration_sec elapses, continue mowing for the rest of the duration; if mow_duration_sec elapses first, dock the mower per existing logic. Document this composition in the docstring.

3. **return_to_dock=True with verify=True**: After Phase 3 succeeds (blade engagement confirmed), then proceed to dock_and_clear semantics (recall + post-dock cancel). If verify fails, STILL try to dock the mower (don't leave it in a partial state) — log the verification failure + attempt dock recovery.

4. **Mower returns to dock mid-Phase-3 NORMALLY**: blade_used_time may not increment by 0.001 hr if the mower mows for <3.6 sec, but the broader sequence "mowed area complete" is success. Per pseudocode §3: when state=docked and elapsed > 120s, check final blade_used_time delta — if ≥ 0.001, return success; otherwise return failure.

5. **HA connectivity drops mid-polling**: each `get_ha_state` call inside `_verify_mowing` should use the existing retry-on-NetworkError logic in ha_client. If retries exhaust → raise RuntimeError "HA connectivity lost during verification" (NOT a verification failure; an environmental failure).

6. **The opt-out path (verify=False)**: returns `result="mow_dispatched_unverified"`. Caller must understand they're getting v1.0 semantics + must verify themselves. Document this clearly.
